## ユースケース別の選定

### 大規模アプリ → コンパイル時方式

複数人開発ではクラス名の衝突、不要スタイルの蓄積、トークンのタイポが実行時まで発覚しないといった問題がある。コンパイル時方式の DX 向上効果はツールチェーンのコストを上回る

| 方式 | 適用シーン |
| ------------------- | ------------------------------------- |
| **vanilla-extract** | 最強の型安全性 + テーマコントラクト |
| **Panda CSS** | Tailwind 的な記法 + 型安全性、アプリ開発向け |
| **StyleX** | 大規模アプリ、コンパイル時アトミック化による重複排除 |

::: tip

- 真の価値はトークン検証とデッドコード除去にある
- ハッシュ化クラスによる衝突防止（BEM なら元々衝突しない）は過大評価されがちな利点
- co-location（CSS ファイルもコンポーネントの横に配置できる）は過大評価されがちな利点
:::

---

### 中小規模アプリ → Tailwind

ファイルを切り替えずにスタイルを書け、デザインシステムの制約が組み込まれており、エコシステムが最も成熟している。

TailwindCSS v4 は CSS-first 設定（`@theme` ディレクティブ）に移行し、ツール層がより薄くなった

---

### コンポーネントライブラリ → 素の CSS

- CSS の成果物自体がパブリック API であり、利用側がオーバーライドやデバッグしやすいセマンティックなクラス名が必要
- ツールチェーンの依存を利用側に伝播させるべきではない。素の CSS はゼロ侵入
- CSS 変数でテーマ切替を実現でき、コンパイル時方式の最終成果物と同じ着地点になる

::: warning
npm インストール型のコンポーネントライブラリに Tailwind を使うのはアンチパターンである。利用側が `content` で `node_modules` をスキャンする設定を入れなければならない（shadcn/ui は例外。コピー&ペーストモデルのため）。コアスタイルには inline style ではなく className を使うこと。inline style はオーバーライド不可で、擬似クラス、メディアクエリ、アニメーションに対応できない
:::
