# WebSocket

WebSocket は単一の TCP 接続上で全二重通信を行うプロトコルであり、クライアントとサーバーがいつでも相互にデータをプッシュできる

::: warning 特徴:

- 全二重通信：クライアントとサーバーが同時にデータを送受信でき、チャット、協同編集、オンラインゲームに適している
- 持続的接続：一度のハンドシェイク後に接続を維持し、HTTP の繰り返し接続確立のオーバーヘッドを回避する
- 低レイテンシ：ポーリング不要で、データ変更時に即座にプッシュされる
- バイナリ対応：テキストとバイナリデータ（ArrayBuffer、Blob など）の両方を転送できる

:::

::: danger 制限事項:

- **ステートフル接続**：サーバーは各接続のメモリを維持する必要があり、水平スケーリングが困難。複数サーバー構成ではメッセージミドルウェア（Redis Pub/Sub など）の導入が必要
- **HTTP キャッシュを利用できない**（Cache-Control、ETag など）
- **ブラウザからプロトコル層の Ping を送信できない**：フロントエンドのハートビートはアプリケーション層メッセージで模倣する必要がある
- **同一オリジンポリシーがない**：あらゆる Web ページから接続可能なため、サーバー側で `Origin` の検証が必須
- 双方向リアルタイム通信が不要なケースで WebSocket を使うのはオーバーエンジニアリングであり、大半の「リアルタイム」シーンは SSE で十分

:::

::: info 適用シーン
チャットルーム、オンラインゲーム、協同編集、リアルタイム取引 ―― **双方がいつでもメッセージを送り合う**必要があるシーン
:::

```txt
┏━━━━━━━ HTTP ポーリング（Polling）━━━━━┓
┃                                     ┃
┃  Client ━━ GET /data ━━▶ Server     ┃  → データあれば返却
┃  Client ━━ GET /data ━━▶ Server     ┃  → N 秒ごとに繰り返し
┃  Client ━━ GET /data ━━▶ Server     ┃  → 無駄なリクエスト多数
┃                                     ┃
┃  単方向：クライアントが能動的に要求       ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛


┏━━━━━━━━ WebSocket ━━━━━━━━━━━━━━━━━━┓
┃                                     ┃
┃  Client ━━ HTTP Upgrade ━━▶ Server  ┃  → ハンドシェイク（1回）
┃  Client ◀━━━━ data ━━━━━▶ Server    ┃  → 双方向リアルタイム
┃  Client ◀━━━━ data ━━━━━▶ Server    ┃  → 再接続不要
┃                                     ┃
┃  双方向：サーバーもプッシュ可能          ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
```

## 基礎概念

| 概念                          | 一言での説明                                                                          |
| ----------------------------- | ------------------------------------------------------------------------------------- |
| ハンドシェイク（Handshake）   | クライアントが HTTP Upgrade リクエストを送信し、サーバーが 101 でプロトコル切替を返す |
| フレーム（Frame）             | WebSocket でデータを転送する最小単位。opcode、payload などを含む                      |
| ハートビート（Ping/Pong）     | 定期的に制御フレームを送信して接続の生存を確認する                                    |
| クローズ（Close）             | いずれかの側がクローズフレームを送信し、双方が 4 ウェイハンドシェイクで切断する       |
| サブプロトコル（Subprotocol） | WebSocket 上で取り決めるアプリケーション層プロトコル（STOMP、GraphQL over WS など）   |

---

## ハンドシェイクの流れ

::: warning 核心ポイント：WebSocket のハンドシェイクは HTTP に基づくが、ハンドシェイク完了後は HTTP ではなくなる

- ハンドシェイク段階は**標準 HTTP リクエスト**であるため、HTTP プロキシやロードバランサーを通過できる
- ハンドシェイク成功後、プロトコルは **WebSocket にアップグレード**され、以降の通信は WebSocket フレームで行われ、HTTP メッセージではなくなる
- これが WebSocket の URL が `ws://` や `wss://` であり、`http://` ではない理由である

:::

---

### ハンドシェイクリクエスト（クライアント → サーバー）

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Sec-WebSocket-Protocol: chat, superchat
```

| リクエストヘッダー       | 説明                                                                   |
| ------------------------ | ---------------------------------------------------------------------- |
| `Upgrade: websocket`     | プロトコルを WebSocket にアップグレードすることを要求                  |
| `Connection: Upgrade`    | 接続アップグレードのリクエストであることを示す                         |
| `Sec-WebSocket-Key`      | ランダム生成された Base64 エンコード文字列。サーバー側の身元検証に使用 |
| `Sec-WebSocket-Version`  | WebSocket プロトコルバージョン（固定で `13`）                          |
| `Sec-WebSocket-Protocol` | オプション。クライアントがサポートするサブプロトコルのリスト           |

---

### ハンドシェイクレスポンス（サーバー → クライアント）

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

| レスポンスヘッダー        | 説明                                                                                  |
| ------------------------- | ------------------------------------------------------------------------------------- |
| `101 Switching Protocols` | プロトコル切替成功                                                                    |
| `Sec-WebSocket-Accept`    | サーバーが `Sec-WebSocket-Key` + 固定 GUID を SHA-1 + Base64 で計算した値。偽造防止用 |

::: tip `Sec-WebSocket-Accept` の計算

```
Sec-WebSocket-Accept = Base64(SHA1(Sec-WebSocket-Key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"))
```

これは暗号化のためではなく、サーバーが確かに WebSocket プロトコルを理解していることを証明するためのもの。通常の HTTP サーバーが誤って 101 を返してしまうことを防ぐ
:::

## 接続のライフサイクル

```txt
Client                                    Server
  │                                          │
  │─── HTTP GET (Upgrade: websocket) ──────▶│  ① ハンドシェイクリクエスト
  │◀── HTTP 101 Switching Protocols ────────│  ② ハンドシェイクレスポンス
  │                                          │
  │◀═══════ 双方向データ転送 ═══════════════▶│  ③ 通信フェーズ
  │─── Ping ────────────────────────────────▶│  ④ ハートビート検出
  │◀── Pong ─────────────────────────────────│
  │                                          │
  │─── Close Frame ─────────────────────────▶│  ⑤ 接続クローズ
  │◀── Close Frame ──────────────────────────│
  │                                          │
```

## データフレーム形式

WebSocket は**フレーム（Frame）** を通じてデータを転送する。各フレームの構造は以下の通り：

```txt
 0               1               2               3
 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7 0 1 2 3 4 5 6 7
+-+-+-+-+-------+-+-------------+-------------------------------+
|F|R|R|R| opcode|M| Payload len |    Extended payload length    |
|I|S|S|S| (4)   |A|     (7)     |           (16/64)             |
|N|V|V|V|       |S|             |   (if payload len==126/127)   |
| |1|2|3|       |K|             |                               |
+-+-+-+-+-------+-+-------------+-------------------------------+
|     Masking-key (0 or 4 bytes)                                |
+---------------------------------------------------------------+
|     Payload Data                                              |
+---------------------------------------------------------------+
```

| フィールド    | 説明                                                                                       |
| ------------- | ------------------------------------------------------------------------------------------ |
| `FIN`         | メッセージの最後のフレームかどうか（1 = 最終フレーム）                                     |
| `opcode`      | フレーム種別：`0x1` テキスト、`0x2` バイナリ、`0x8` クローズ、`0x9` Ping、`0xA` Pong       |
| `MASK`        | クライアントから送信するフレームは**必ず**マスク（MASK=1）、サーバーからは**マスクしない** |
| `Payload len` | データ長（7bit / 7+16bit / 7+64bit）                                                       |

::: warning 核心ポイント

- クライアント → サーバーのデータは**必ずマスク**、サーバー → クライアントは**マスクしない**（プロトコルの強制規定）
- 大きなメッセージは複数フレームに分割して転送（フラグメンテーション）され、受信側は `FIN` ビットでメッセージの完全性を判断する
- `Ping/Pong` は制御フレームであり、ハートビート検出に使用される。業務データは含まない

:::

## ハートビート機構（Ping/Pong）

WebSocket 接続はネットワーク異常により「半死状態」（TCP 接続は存在するが実際には到達不能）になる可能性がある。ハートビート検出はこのような死んだ接続を発見してクリーンアップするために使用される

```txt
正常な場合：
Client ─── Ping ──▶ Server
Client ◀── Pong ─── Server     ← 接続は生存中

異常な場合：
Client ─── Ping ──▶ Server
（タイムアウト待ち、Pong 応答なし）  ← 接続は死んでいる、能動的に切断
```

::: warning 核心ポイント

- Ping/Pong は WebSocket **プロトコル層**の制御フレームであり、アプリケーション層で `{ type: "ping" }` メッセージを自前で送るものではない
- Ping を受信したら **必ず** Pong で応答する（ブラウザは自動処理するが、Node.js の ws ライブラリではライブラリの組み込み動作に依存するか手動処理が必要）
- ハートビート間隔は通常 30〜60 秒。短すぎると帯域の無駄、長すぎると検出が遅れる
- **ブラウザの制限**：ブラウザの `WebSocket` API ではプロトコル層の Ping フレームを**送信できない**。そのため、フロントエンドでは通常アプリケーション層メッセージ（例：`{ type: "ping" }`）でハートビートを模倣し、サーバー側で対応する応答を返す必要がある

:::

- サーバー側（Node.js）：プロトコル層の `ws.ping()` を使用 ―― [Node.js 実装 - ハートビート検出](/ja/programming/web-backend/websocket/sample-node#ハートビート検出)を参照
- ブラウザ側（React）：アプリケーション層メッセージで模倣 ―― [React 実装 - useWebSocket](/ja/programming/web-backend/websocket/sample-react#usewebsocket-カスタム-hook)を参照

## 基本的な使い方

サーバー側ではよく [ws](https://github.com/websockets/ws) ライブラリ（ネイティブプロトコル）や [Socket.IO](https://socket.io/)（高レベルラッパー）が使われる。詳細は [Node.js 実装](/ja/programming/web-backend/websocket/sample-node)を参照

ブラウザ側ではネイティブの `WebSocket` API を使用する：

```js
const ws = new WebSocket("ws://localhost:3000");

ws.onopen = () => {
  ws.send("Hello"); // テキストを送信
  ws.send(new Blob(["binary data"])); // バイナリを送信
  ws.send(JSON.stringify({ type: "join", room: 1 })); // JSON を送信
};

ws.onmessage = (event) => {
  console.log("受信:", event.data); // string または Blob
};

ws.onclose = (event) => {
  console.log(`クローズ: code=${event.code}, reason=${event.reason}`);
  // event.wasClean は正常クローズかどうかを示す
};

ws.onerror = (error) => {
  console.error("エラー:", error);
};

// readyState: 0=CONNECTING, 1=OPEN, 2=CLOSING, 3=CLOSED
console.log(ws.readyState);

// 能動的にクローズ
ws.close(1000, "Normal closure");
```

::: warning `readyState` の 4 つの状態

| 値  | 定数                   | 意味                                 |
| --- | ---------------------- | ------------------------------------ |
| `0` | `WebSocket.CONNECTING` | 接続中、まだ確立されていない         |
| `1` | `WebSocket.OPEN`       | 接続確立済み、通信可能               |
| `2` | `WebSocket.CLOSING`    | クローズ中、クローズフレーム送信済み |
| `3` | `WebSocket.CLOSED`     | クローズ済みまたは接続失敗           |

データ送信前に**必ず** `readyState === 1` を確認すること。そうでないと例外がスローされる
:::

## セキュリティ

### wss:// 暗号化

`ws://` は平文通信であり、本番環境では必ず `wss://`（WebSocket over TLS）を使用する。HTTPS と HTTP の関係と同じである

```txt
ws://example.com/chat     → 平文、安全でない
wss://example.com/chat    → TLS 暗号化、本番環境では必須
```

---

### よくあるセキュリティ問題

| 問題                                         | 説明                                                                                                 | 防御策                                                                                            |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| クロスサイト WebSocket ハイジャック（CSWSH） | 悪意のあるページが WS サーバーに接続し、ブラウザが自動的に Cookie を送信する                         | `Origin` リクエストヘッダーを検証し、ホワイトリスト外のオリジンを拒否                             |
| 認証                                         | WebSocket のハンドシェイクは HTTP だが、その後は HTTP ではなくなるため、毎回 Cookie/Token を送れない | ハンドシェイク段階で検証（URL パラメータ、Cookie）、または接続後の最初のメッセージで Token を送信 |
| DoS                                          | 悪意のあるクライアントが大量の接続や巨大なメッセージを送信する                                       | 同時接続数の制限、メッセージサイズの制限、レート制限                                              |

::: warning 核心ポイント：WebSocket には同一オリジンポリシーがない

HTTP には CORS 制限があるが、WebSocket には**同一オリジンポリシーがない**。あらゆる Web ページが任意の WebSocket サーバーに接続を開始できる。そのため**サーバー側で `Origin` ヘッダーを検証することが必須**である
:::

## クローズステータスコード

| ステータスコード | 意味                   | よくあるケース                                       |
| ---------------- | ---------------------- | ---------------------------------------------------- |
| `1000`           | 正常クローズ           | 能動的に `ws.close()` を呼び出し                     |
| `1001`           | エンドポイント離脱     | ページ遷移、サーバーシャットダウン                   |
| `1002`           | プロトコルエラー       | プロトコルに違反するフレームを受信                   |
| `1003`           | データ型エラー         | サポートしていないデータ型を受信                     |
| `1006`           | 異常クローズ           | クローズフレームを受信していない（ネットワーク切断） |
| `1008`           | ポリシー違反           | メッセージがサーバーのポリシーに違反                 |
| `1009`           | メッセージが大きすぎる | サーバーの最大メッセージ制限を超過                   |
| `1011`           | 予期しないエラー       | サーバーが予期しない状況に遭遇                       |

> `1006` はコードで手動送信できず、接続が異常切断された場合にのみブラウザが自動で設定する

## スケーラビリティの問題

WebSocket アプリケーションを複数サーバーにスケールアウトする際、核心的な問題に直面する：

```txt
ユーザー A は Server 1 に接続
ユーザー B は Server 2 に接続

ユーザー A がユーザー B にメッセージを送信 → メッセージは Server 1 にしかなく、Server 2 には届かない
```

---

### 解決策：メッセージミドルウェア

```txt
Server 1 ──publish──▶ Redis Pub/Sub ──subscribe──▶ Server 2
                          │
                          ▼
                      Server 3, 4, ...
```

よくある方式：

| 方式                                     | 説明                                                                                                   |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Redis Pub/Sub                            | 最も一般的でシンプルかつ高効率。Socket.IO には公式 Redis Adapter がある                                |
| Kafka / RabbitMQ                         | メッセージの永続化や複雑なルーティングが必要な場合に使用                                               |
| スティッキーセッション（Sticky Session） | ロードバランサーが同一ユーザーを常に同じサーバーにルーティングするが、サーバー間通信の問題は解決しない |

::: warning 核心ポイント

- 単一サーバーの WebSocket 接続数はメモリとファイルディスクリプタに制限され、上限は通常**数万〜10 万程度**
- 水平スケーリング時には**必ずメッセージミドルウェアを導入**する必要がある。そうしないと異なるサーバー上のユーザー間で通信できない
- Socket.IO は `@socket.io/redis-adapter` を提供しており、すぐに使えるソリューションとなる

:::
