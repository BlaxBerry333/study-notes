# API 通信方式

## データリクエスト方式

4つの方式がそれぞれ異なる課題を解決する。まず選定の結論を示す：

::: warning どう選ぶか

- **外部公開 API / シンプルな CRUD** → **REST**（最も成熟、エコシステムが充実）
- **複数クライアントで柔軟にクエリ（Web + Mobile で API 共用）** → **[GraphQL](/ja/programming/web-backend/graphql/)**
- **TypeScript フルスタックの内部プロジェクト** → **[tRPC](/ja/programming/web-backend/trpc/)**（ゼロ設定で型安全、開発速度最速）
- **マイクロサービス間通信 / 高性能 / 多言語間呼び出し** → **[gRPC](/ja/programming/web-backend/grpc/)**

:::

---

### REST

最も伝統的な方式。各リソースに1つの URL を割り当て、HTTP メソッド（GET/POST/PUT/DELETE）で操作を表現する

- インターフェース定義：URL + HTTP メソッド（`GET /api/users/1`）
- 型安全：追加ツールが必要（OpenAPI + codegen）
- キャッシュ：HTTP ネイティブキャッシュ + CDN、エコシステムが最も成熟
- 用途：公開 API、シンプルな CRUD、HTTP キャッシュが必要な場面

---

### [GraphQL](/ja/programming/web-backend/graphql/)

クライアントが必要なフィールドを正確に指定し、1回のリクエストで関連データをすべて取得する

- インターフェース定義：Schema（SDL）+ 単一エンドポイント（`POST /graphql`）
- 型安全：Schema に型が内包されており、codegen でクライアント型を生成可能
- キャッシュ：HTTP キャッシュ非対応、クライアント側の仕組みが必要（Apollo Cache）
- 用途：複数クライアントで API 共用、データ関係が複雑、フロントエンドで柔軟に組み合わせたい場合

---

### [tRPC](/ja/programming/web-backend/trpc/)

クライアントからサーバー側の関数を直接呼び出し、TypeScript コンパイラが自動的に型を推論する

- インターフェース定義：TypeScript の関数そのもの、中間層なし
- 型安全：**ゼロ設定でエンドツーエンドの型安全**、Schema や codegen 不要
- キャッシュ：TanStack Query に依存
- 制限：**TypeScript 限定**、非 TS クライアントには提供不可

---

### [gRPC](/ja/programming/web-backend/grpc/)

HTTP/2 + Protocol Buffers ベースの高性能 RPC フレームワーク

- インターフェース定義：`.proto` ファイル（強い型の契約）、codegen で多言語コードを生成
- 型安全：[protobuf](/ja/programming/web-backend/protobuf/) codegen による多言語型安全
- キャッシュ：HTTP キャッシュ非対応
- 用途：マイクロサービス内部通信、高スループット低レイテンシ、ストリーミング、多言語サービス間呼び出し

---

### 比較早見表

| 観点       | REST                       | GraphQL                        | tRPC                     | gRPC                    |
| ---------- | -------------------------- | ------------------------------ | ------------------------ | ----------------------- |
| データ取得 | サーバー側が返却構造を決定 | クライアントがフィールドを指定 | サーバー側関数を呼び出し | Stub メソッドを呼び出し |
| 多言語対応 | 任意                       | 任意                           | TS のみ                  | 10+ 言語                |
| キャッシュ | HTTP ネイティブ            | クライアント側の仕組みが必要   | TanStack Query           | 非対応                  |
| ブラウザ   | ネイティブ対応             | ネイティブ対応                 | ネイティブ対応           | gRPC-Web プロキシが必要 |

---

## リアルタイム通信の選定

::: warning どう選ぶか

- ほとんどの「リアルタイム」要件は実は **SSE** で十分 --- 通知プッシュ、リアルタイムダッシュボード、AI ストリーミング出力はすべてサーバーからの一方向プッシュであり、クライアントからのデータ送信は通常の HTTP リクエストで問題ない
- **双方向でいつでもメッセージを送り合う**必要がある場面（チャットルーム、オンラインゲーム、協同編集）でのみ **WebSocket** が本当に必要になる
- **HTTP ポーリング**はインフラが長時間接続に対応していない場合やリアルタイム性の要求が極めて低い場合にのみ使用する

:::

---

### HTTP ポーリング

> Polling

クライアントが N 秒ごとにリクエストを送って「新しいデータはあるか」と問い合わせる方式。最もシンプルだが最も無駄が多い

- 通信方向：クライアント主導（単方向）
- 実装の複雑度：低
- 用途：互換性要求が高い、リアルタイム性の要求が低い場合

---

### [SSE](/ja/programming/web-backend/sse/)

> Server-Sent Events

サーバーがクローズしない HTTP レスポンスを通じて継続的にデータをプッシュする

- 通信方向：サーバー → クライアント（単方向）
- プロトコル：通常の HTTP（`http://` / `https://`）
- ブラウザがネイティブで自動再接続と中断復帰（`Last-Event-ID`）をサポート
- データ形式：テキストのみ
- 用途：通知プッシュ、ログストリーム、AI ストリーミング出力

---

### [WebSocket](/ja/programming/web-backend/websocket/)

クライアントとサーバーがプロトコルアップグレードにより全二重接続を確立し、双方がいつでもメッセージを送信できる

- 通信方向：全二重（双方向）
- プロトコル：`ws://` / `wss://`（HTTP ハンドシェイク後にアップグレード）
- 再接続ロジックは手動実装が必要
- データ形式：テキスト + バイナリ
- 用途：チャット、ゲーム、協同編集

---

### 比較早見表

| 観点                   | HTTP ポーリング          | SSE                     | WebSocket                                 |
| ---------------------- | ------------------------ | ----------------------- | ----------------------------------------- |
| 通信方向               | クライアント主導         | サーバー → クライアント | 全二重                                    |
| 接続方式               | 短時間接続               | 長時間接続（HTTP）      | 長時間接続（プロトコルアップグレード）    |
| ヘッダーオーバーヘッド | 毎回完全な HTTP ヘッダー | 初回のみ                | ハンドシェイク後 2~14 バイト              |
| 自動再接続             | 該当なし                 | ブラウザネイティブ      | 手動実装が必要                            |
| クロスオリジン         | CORS                     | CORS                    | 同一オリジン制限なし（Origin 検証が必要） |
| バイナリ               | 非対応                   | 非対応                  | 対応（ArrayBuffer / Blob）                |
| スケーラビリティ       | ステートレス、天然拡張可 | HTTP インフラで対応可   | ステートフル、Redis Pub/Sub 等が必要      |

::: tip ゲーム連携サービスの選定

ゲームの場面では通常、複数のリアルタイム要件があり、それぞれに適した方式がある：

| 要件 | 方式 | 理由 |
| --- | --- | --- |
| ゲーム状態同期（操作 ↔ サーバー） | **WebSocket** | 双方向低遅延、バイナリデータ対応 |
| ランキング / イベント告知プッシュ | **SSE** | サーバーからの一方向プッシュ、HTTP インフラと自然に互換 |
| フレンドのオンライン状態 | **WebSocket** | 双方向（自分の状態を報告 + 友達の状態変化を受信） |
| 履歴 / 実績の照会 | **REST / tRPC** | 通常のデータリクエスト、リアルタイム不要 |

全ての要件に WebSocket を使わない——運用コストが高く、スケーリングも複雑。**要件ごとに分離：リアルタイム双方向は WebSocket、一方向プッシュは SSE、通常クエリは HTTP**

:::
