# SSE

SSE（Server-Sent Events）は HTTP ベースのサーバーからの一方向プッシュ技術であり、サーバーがクライアントに対して継続的にイベントストリームを送信できる

::: warning 特徴:

- 一方向通信：サーバーからクライアントへのプッシュのみ。通知、ログストリーム、AI ストリーミング出力に適している
- HTTP ベース：標準 HTTP プロトコルを使用し、プロトコルのアップグレードが不要。プロキシやファイアウォールとネイティブに互換性がある
- 自動再接続：ブラウザがネイティブに断線自動再接続をサポートし、`Last-Event-ID` を携帯して復旧する
- テキストプロトコル：データ形式は純粋なテキスト（`text/event-stream`）であり、軽量でデバッグしやすい
:::

::: danger 制限事項:

- **一方向通信**：サーバーからのプッシュのみで、クライアントのデータ送信には別途 HTTP リクエストが必要
- **EventSource は GET のみサポート**：body を送信できない。POST が必要な場合は fetch + ReadableStream で代替する必要がある
- **カスタムリクエストヘッダーをサポートしない**：`Authorization` を直接送信できず、URL パラメータまたは fetch で代替する必要がある
- **テキストのみ**：バイナリデータをサポートしない
- **HTTP/1.1 の接続数制限**：ドメインあたり最大 6 つの SSE 接続。HTTP/2 の多重化で解決する必要がある
:::

::: info 適用シーン
通知プッシュ、リアルタイムデータパネル、ログストリーム、AI ストリーミング出力 ―― **サーバーからの一方向プッシュ**のシーン。クライアントのデータ送信は通常の HTTP リクエストで行う。選定比較は [API 通信方式](/ja/programming/web-backend/api-communication#リアルタイム通信の選定)を参照
:::

## 基礎概念

| 概念         | 一言での説明                                         |
| ------------ | ---------------------------------------------------- |
| EventSource  | ブラウザのネイティブ API。SSE イベントストリームの受信に使用 |
| event stream | サーバーが返す `text/event-stream` 形式のデータストリーム |
| event        | イベント名。クライアントは `addEventListener` で特定イベントを監視 |
| data         | イベントが運ぶデータ。複数行をサポート               |
| id           | イベント ID。断線再接続時に `Last-Event-ID` リクエストヘッダーで復旧 |
| retry        | サーバーが指定するクライアントの再接続間隔（ミリ秒） |

## 動作原理

SSE は本質的に**終了しない HTTP レスポンス**である：

```txt
Client                                       Server
  │                                             │
  │─── GET /api/events ────────────────────────▶│  ① 通常の HTTP リクエスト
  │◀── 200 OK                                   │  ② レスポンスヘッダー：Content-Type: text/event-stream
  │    Content-Type: text/event-stream           │
  │                                             │
  │◀── data: {"msg": "hello"}\n\n ──────────────│  ③ イベントを継続的にプッシュ
  │◀── data: {"msg": "world"}\n\n ──────────────│     （レスポンスはクローズしない）
  │◀── data: {"msg": "..."}\n\n ────────────────│
  │                                             │
  │    （接続切断）                               │  ④ ネットワーク異常
  │                                             │
  │─── GET /api/events ────────────────────────▶│  ⑤ ブラウザが自動再接続
  │    Last-Event-ID: 3                          │     前回のイベント ID を送信
  │◀── 200 OK ...                               │  ⑥ サーバーが断点から復旧
```

::: warning 核心ポイント：SSE は HTTP であり、新しいプロトコルではない

- プロトコルのアップグレードは不要（`101 Switching Protocols` がない）
- サーバーは正しいレスポンスヘッダーを設定し、**レスポンスをクローズせずに**データを書き込み続けるだけ
- すべての HTTP インフラ（プロキシ、CDN、ロードバランサー）がそのまま対応できる
- これが SSE が WebSocket より**はるかにシンプル**な理由である
:::

## プロトコル形式

SSE は純粋なテキスト形式を使用する。各フィールドが 1 行を占め、イベント間は空行で区切られる

```txt
event: message
data: {"text": "Hello"}
id: 1

event: notification
data: 新着メッセージ通知
id: 2

: これはコメントで、無視される（ハートビートキープアライブによく使用される）

data: event フィールドがない場合、デフォルトの message イベントが発火する
id: 3

retry: 5000
data: 再接続間隔を 5 秒に設定
```

---

### フィールド説明

| フィールド | 説明                                         | 必須かどうか     |
| ---------- | -------------------------------------------- | ---------------- |
| `data`     | イベントデータ。複数行可能（最終的に `\n` で結合） | はい             |
| `event`    | イベント種別名。省略するとデフォルトの `message` | いいえ           |
| `id`       | イベント ID。断線再接続時に `Last-Event-ID` で復旧 | いいえ（ただし強く推奨） |
| `retry`    | 再接続間隔（ミリ秒）。クライアントの再接続タイミングを制御 | いいえ           |
| `: コメント` | コロンで始まる行は無視される                 | ―                |

::: warning 形式のポイント

- 各フィールドの形式は `field: value`（コロンの後にスペース 1 つ）
- `data` は複数行が可能で、最終的に `\n` で結合される
- イベント間は**空行**（`\n\n`）で区切る。これがイベント境界のマーカーとなる
- `:` で始まる行はコメントで、**ハートビートキープアライブ**によく使われる（プロキシのタイムアウトによる切断を防ぐ）
:::

---

### 複数行 data の例

```txt
data: 1 行目
data: 2 行目
data: 3 行目

```

クライアントが受信する `event.data` は `"1 行目\n2 行目\n3 行目"` となる

## 基本的な使い方

### ブラウザ側（EventSource API）

```js
// 接続を作成
const es = new EventSource("/api/events");

// デフォルトの message イベント（event フィールドのないイベント）
es.onmessage = (event) => {
  console.log("受信:", event.data);
  console.log("ID:", event.lastEventId);
};

// カスタムイベントを監視
es.addEventListener("notification", (event) => {
  const data = JSON.parse(event.data);
  console.log("通知:", data);
});

// 接続確立
es.onopen = () => {
  console.log("接続確立");
};

// エラーハンドリング
es.onerror = () => {
  // 手動で再接続する必要はなく、ブラウザが自動処理する
  if (es.readyState === EventSource.CLOSED) {
    console.log("接続が永久にクローズされた");
  } else {
    console.log("接続が中断、ブラウザが再接続中...");
  }
};

// 能動的にクローズ（呼び出すと自動再接続しない）
es.close();
```

::: warning `readyState` の 3 つの状態

| 値  | 定数                     | 意味                       |
| --- | ------------------------ | -------------------------- |
| `0` | `EventSource.CONNECTING` | 接続中 / 再接続中          |
| `1` | `EventSource.OPEN`       | 接続確立済み               |
| `2` | `EventSource.CLOSED`     | 接続クローズ済み（再接続しない） |

`close()` メソッドまたはサーバーが非 200 / 非 `text/event-stream` を返した場合のみ `CLOSED` 状態になる。ネットワーク中断では `CONNECTING`（再接続中）になるだけである
:::

---

### サーバー側の実装ポイント

```js
app.get("/api/events", (req, res) => {
  // 3 つの必要なレスポンスヘッダー
  res.writeHead(200, {
    "Content-Type": "text/event-stream", // ブラウザに SSE であることを伝える
    "Cache-Control": "no-cache", // キャッシュを禁止
    Connection: "keep-alive", // 接続を維持
  });

  // イベントをプッシュ...
});
```

完全な実装例は [Node.js 実装](/ja/programming/web-backend/sse/sample-node)を参照

## 断線再接続機構

ブラウザが断線再接続をネイティブサポートし、断点からの復旧も可能：

---


### 動作フロー

```txt
① サーバーがイベント送信時に id を付与
   → id: 42\ndata: {...}\n\n

② 接続切断（ネットワーク異常）

③ ブラウザが retry 時間の後に自動再接続（デフォルト約 3 秒）

④ 再接続リクエストに自動でリクエストヘッダー Last-Event-ID: 42 が付与される

⑤ サーバーが Last-Event-ID を読み取り、43 からプッシュを継続
```

::: warning 核心ポイント

- `id` フィールドは断点復旧の基盤であり、**id を送信しなければ復旧できない**
- `retry` フィールドで再接続間隔を制御できる。単位はミリ秒（`retry: 5000` = 5 秒）
- 再接続時にブラウザが自動でリクエストヘッダーに `Last-Event-ID` を付与する。手動処理は不要
- サーバーは `Last-Event-ID` に基づいて断点を見つけられる必要がある（メモリキュー、データベース、Redis など）
- `close()` でクローズした接続は**再接続しない**
:::

実装例は [Node.js 実装 - 断線再接続](/ja/programming/web-backend/sse/sample-node#断線再接続と-last-event-id)を参照

## 実際の応用シーン

### AI ストリーミングレスポンス（ChatGPT のような）

AI 対話インターフェースは SSE の最も典型的な現代の応用シーンである：

```txt
Client ─── POST /api/chat (通常の HTTP リクエスト) ──▶ Server
Client ◀── text/event-stream                   Server

data: {"token": "こ"}
data: {"token": "ん"}
data: {"token": "に"}
data: {"token": "ち"}
data: {"token": "は"}
data: {"token": "、"}
data: {"token": "AI"}
data: {"token": "で"}
data: {"token": "す"}
data: [DONE]
```

> 注意：ここで実際に使われているのは `POST` + `fetch` + `ReadableStream` であり、`EventSource` ではない（body の送信が必要なため）。ただし**データ形式は SSE プロトコルに準拠**している

---

### その他のよくあるシーン

| シーン          | 説明                                     |
| --------------- | ---------------------------------------- |
| 通知プッシュ    | 新着メッセージ、システムアナウンス。サーバーに通知があればプッシュ |
| リアルタイムデータパネル | 監視ダッシュボード、株価。定期的に最新データをプッシュ |
| ビルド/デプロイログ | CI/CD ログをリアルタイムでブラウザに出力 |
| プログレスバー  | 長時間タスク（アップロード、トランスコード、エクスポート）の進捗フィードバック |
