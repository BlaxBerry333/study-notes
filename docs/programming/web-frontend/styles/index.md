## 场景选型

### 大型应用 → 编译时方案

多人协作时 class 命名冲突、废弃样式堆积、token 拼错运行时才发现——编译时方案的 DX 收益远大于工具链成本

| 方案                | 适用场景                              |
| ------------------- | ------------------------------------- |
| **vanilla-extract** | 最强类型安全 + 主题契约               |
| **Panda CSS**       | 类 Tailwind 写法 + 类型安全，应用开发 |
| **StyleX**          | 大规模应用，编译时原子化去重          |

::: tip

- 真正的价值在 token 校验和死代码消除
- hash class 防冲突（BEM 本来就不冲突）是被高估的卖点
- co-location（CSS 文件也能放组件旁边）是被高估的卖点
:::

---

### 中小型应用 → Tailwind

不用切换文件写样式，设计系统约束内建，生态最成熟。

TailwindCSS v4 转向 CSS-first 配置（`@theme` 指令），工具层更薄

---

### 组件库 → 纯 CSS

- CSS 产物本身是 public API，需要语义化 class name 让消费方覆盖和调试
- 不应把工具链依赖传递给消费方，纯 CSS 零侵入
- CSS 变量实现主题切换，和编译时方案的最终产物殊途同归

::: warning
npm 安装型组件库用 Tailwind 是反模式——消费方必须配置 `content` 扫描 `node_modules`（shadcn/ui 例外，它是 copy-paste 模式）。核心样式用 className 而非 inline style——inline style 无法被覆盖，不支持伪类、媒体查询、动画
:::
